## Q1: IExecute 인터페이스가 반드시 필요한가?

**질문:**
`IExecute` 인터페이스가 프로젝트 구조상 꼭 필요한 존재인가?

**답변:**
결론적으로 **추상화와 의존성 분리를 위해 필수적**입니다. 
1. **플랫폼과 로직의 분리**: OS 진입점(`WinMain` 등)이 특정 앱 로직을 직접 알지 못하게 하여 플랫폼 코드를 범용적으로 유지할 수 있습니다.
2. **다중 플랫폼 대응**: `ExecWindows`, `ExecAndroid` 등 서로 다른 플랫폼이 동일한 인터페이스(`OnUpdate`, `OnRender` 등)를 통해 앱을 실행할 수 있게 합니다.
3. **제어의 역전(IoC)**: 플랫폼이 루프의 주도권을 갖고, 앱은 호출되는 타이밍에 맞춰 로직만 수행하게 함으로써 구조적 안정성을 확보합니다.

## Q2: Execute 클래스는 Engine, Application, Platform 중 어디에 있어야 하는가?

**질문:**
실제 구현체인 `Execute` 클래스는 어디에 위치해야 하는가?

**답변:**
각 레이어의 역할에 따라 다음과 같이 구분하는 것이 이상적입니다.
1. **Engine**: `IExecute` 인터페이스(규격)만 소유합니다.
2. **Application**: 실제 앱의 비즈니스 로직(게임 로직 등)을 담은 구현체가 존재합니다. (가장 적절한 위치)
3. **Platform**: OS별 시스템 루프를 수행하며, Application에서 만든 구현체를 넘겨받아 실행(Run)해주는 Runner 역할을 합니다.

현재는 플랫폼 종속적인 초기화 때문에 일시적으로 `Platform` 쪽에 있을 수 있으나, 최종적으로는 실제 콘텐츠 로직을 담은 `Execute`는 `Application` 레이어로 옮겨지는 것이 맞습니다.

## Q3: 엔진 배포 및 Application 교체 전략

**질문:**
Application이 클라이언트 역할로 매번 바뀔 예정인데, 엔진 배포는 어떤 방식이 좋은가?

**답변:**
현대적인 엔진들처럼 **SDK 방식의 배포**를 권장합니다.
1. **Engine**: 빌드된 라이브러리(`Engine.lib`)와 공개 헤더 파일들을 묶어 SDK 형태로 배포됩니다.
2. **Application**: 사용자가 엔진 SDK를 참조하여 자신만의 로직을 구현하는 독립적인 프로젝트입니다.
3. **Runner (Platform)**: 엔진 라이브러리와 Application 구현체를 링크하여 최종 실행 파일(`.exe`, `.apk`)을 생성하는 템플릿 프로젝트 형태로 제공됩니다.

이 방식을 사용하면 엔진 코드를 수정하지 않고도 `Application` 프로젝트만 교체하여 새로운 앱을 빠르게 개발할 수 있습니다.
