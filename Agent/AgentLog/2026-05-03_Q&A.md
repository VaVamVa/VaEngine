## Q14: 파생 클래스에서 virtual 제거와 override 사용
**질문:**
가상 함수를 상속받을 때 `virtual`을 지우고 `override`만 남기는 것의 의미.

**답변:**
파생 클래스에서 `virtual`은 생략해도 동작이 동일함. `virtual`은 "새 가상 함수 도입"의 의미이고, `override`는 "기반 클래스의 가상 함수를 재정의한다"는 컴파일러 강제 계약임. `virtual`을 같이 쓰면 새로 선언하는 것인지 재정의하는 것인지 혼동을 줄 수 있음.

`override` 없이 `virtual`만 쓰면 기반 클래스 시그니처가 바뀌어도 파생 클래스가 몰래 새 가상 함수가 되어버리는 버그가 생김. C++11 이후 파생 클래스에서는 `override`만 사용하는 것이 권장 스타일.

## Q15: Private 헤더 IDE 내비게이션 문제와 FILE_SET HEADERS
**질문:**
`Common_DirectX.h`가 빌드는 되는데 F12/Ctrl+Click 내비게이션이 안 됨.

**답변:**
`FILE_SET HEADERS`가 `Public/*.h`에만 등록되어 있어서 IDE가 Private 헤더를 프로젝트 파일로 인식하지 못하기 때문. 컴파일러는 include 경로와 파일시스템으로 헤더를 찾지만, IDE는 CMake가 알려준 파일 목록 기반으로 탐색함.

Private 헤더도 별도 이름의 `FILE_SET`으로 등록하면 IDE 내비게이션이 해결됨. 빌드에는 영향 없음.

```cmake
target_sources(Engine PRIVATE
    FILE_SET nav_DX12_HEADERS TYPE HEADERS
    BASE_DIRS Private
    FILES ${ENGINE_DX12_HEADERS}
)
```

## Q16: CMake에서 .cpp만 GLOB하는 이유
**질문:**
구현부 `.cpp`만 GLOB으로 수집하고 헤더는 수집하지 않는 이유.

**답변:**
CMake에서 헤더 파일은 컴파일 대상이 아님. `target_sources`는 컴파일러에게 번역 단위를 알리는 것이고, 헤더는 `target_include_directories`로 경로만 알려주면 컴파일러가 `#include` 시점에 알아서 찾음.

`.cpp`만 GLOB하는 것은 빌드 정확성 측면에서 올바른 CMake 관례임. Private 헤더에 `FILE_SET`을 추가하는 것은 순수히 IDE 내비게이션을 위한 별도 작업.

## Q17: FILE_SET HEADERS PRIVATE 스코프 충돌 오류
**질문:**
`target_sources Scope PRIVATE for file set "HEADERS" does not match original scope PUBLIC` 오류.

**답변:**
`FILE_SET HEADERS`는 이름을 지정하지 않으면 기본 이름 `"HEADERS"`를 공유함. Public 헤더가 이미 `PUBLIC` 스코프로 `"HEADERS"` 세트를 선언했기 때문에 같은 이름에 `PRIVATE`를 쓰면 충돌 발생.

별도 이름을 부여하면 독립적인 HEADERS 세트로 등록됨. FILE_SET 이름은 소문자로 시작해야 하며, `HEADERS`는 CMake 내장 예약 이름이라 예외적으로 대문자.

```cmake
target_sources(Engine PRIVATE
    FILE_SET nav_DX12_HEADERS TYPE HEADERS
    BASE_DIRS Private
    FILES ${ENGINE_DX12_HEADERS}
)
```

## Q18: DirectX-Headers CMake 관리 — FetchContent vs Git Submodule
**질문:**
ThirdParty 라이브러리를 git에 직접 커밋하지 않고 CMake로 관리하는 방법.

**답변:**
두 가지 방법이 있음.

**FetchContent** (권장): CMake 빌드 구성 시점에 빌드 디렉토리 내 `_deps/`에 자동 다운로드. `GIT_TAG`로 버전 고정. 팀원 추가 작업 불필요.
```cmake
include(FetchContent)
FetchContent_Declare(DirectXHeaders GIT_REPOSITORY ... GIT_TAG v1.614.1)
FetchContent_MakeAvailable(DirectXHeaders)
target_link_libraries(Engine PRIVATE Microsoft::DirectX-Headers)
```

**Git Submodule**: 소스 트리 내에 위치하나 파일 자체는 추적하지 않고 커밋 해시만 기록. `git submodule update --init` 으로 취득. 클론 시 `--recurse-submodules` 필요.

혼자 개발하는 헤더 전용 라이브러리라면 FetchContent가 설정이 더 단순함.

## Q20: d3dx12.h 경로 미인식 — target_include_directories 오용
**질문:**
`<d3dx12.h>`를 `<directx/d3dx12.h>`로 수정했는데도 인식 안 됨.

**답변:**
두 가지 원인이 있었음.

1. **include 경로**: 기존에는 `include/directx/`를 직접 지정했기 때문에 `<d3dx12.h>`가 동작했으나, `Microsoft::DirectX-Headers` 타겟은 base를 `include/`로 노출하므로 `<directx/d3dx12.h>`로 변경해야 함.

2. **잘못된 CMake 명령 사용**: `target_include_directories`는 실제 디렉토리 경로를 받는 명령임. CMake 타겟 이름을 여기에 넣어도 경로로 인식하지 않음. CMake 타겟의 include 경로를 전파받으려면 `target_link_libraries`에 추가해야 함.

```cmake
# 잘못된 방식
target_include_directories(Engine PRIVATE Microsoft::DirectX-Headers)

# 올바른 방식
target_link_libraries(Engine PRIVATE d3d12 dxgi dxguid Microsoft::DirectX-Headers)
```

## Q21: DirectX-Headers의 d3d12.h 사용 가능 여부
**질문:**
DirectX-Headers 내부의 `d3d12.h`를 사용해도 되는가?

**답변:**
사용 가능함. DirectX-Headers의 `d3d12.h`는 Microsoft가 직접 관리하는 공식 헤더로 Windows SDK의 것과 동일한 소스이며 오히려 더 빠르게 업데이트됨. `<directx/d3d12.h>`로 포함.

단, `dxgi1_6.h`는 DirectX-Headers에 포함되어 있지 않으므로 Windows SDK 경로(`<dxgi1_6.h>`)를 유지해야 함. DXGI는 Windows 전용 API라 크로스플랫폼 대상이 아니기 때문.

## Q22: DXGI FetchContent 불가 이유
**질문:**
DXGI 헤더도 FetchContent로 가져올 방법이 있는가?

**답변:**
없음. DXGI는 Windows 전용 API라서 DirectX-Headers에 포함되어 있지 않음. DirectX-Headers의 주요 목적 중 하나가 D3D12 헤더를 Linux/WSL에서도 쓸 수 있게 하는 것인데, DXGI 자체가 Windows에만 존재하므로 크로스플랫폼화할 이유가 없음. `<dxgi1_6.h>`는 Windows SDK에서 가져오는 것이 유일한 방법.

## Q23: Vulkan FetchContent 방법
**질문:**
Vulkan도 FetchContent로 가져올 수 있는가?

**답변:**
가능하나 현재 프로젝트에서는 권장하지 않음. `Vulkan-Headers`(KhronosGroup)를 FetchContent로 가져올 수 있지만 헤더만 제공되므로 `vulkan-1.lib` 링크는 별도로 필요함.

Vulkan SDK는 드라이버 레이어, 유효성 검사 레이어, `glslc` 등을 포함하는 통합 패키지이므로 시스템에 SDK를 설치하고 `find_package(Vulkan REQUIRED)`를 사용하는 것이 표준 방식임. Android는 NDK에 Vulkan이 내장되어 있어 별도 작업 불필요.
