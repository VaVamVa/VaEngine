## Q1: ExecAndroid에서 Engine/Application 참조 경고

**질문:**
ExecAndroid 프로젝트에서 Engine과 Application이 지원되지 않는 참조 항목이라는 경고가 뜸.

**답변:**

### 결론
정상 동작. ExecAndroid는 MSBuild 프로젝트 참조를 사용할 수 없는 구조이므로 경고가 맞으며, 무시해도 됨.

### 원인

| | ExecWindows | ExecAndroid |
|---|---|---|
| 빌드 시스템 | MSBuild | Gradle + Android NDK |
| 컴파일러 | MSVC (x64) | Clang (ARM/ARM64) |
| 참조 방식 | `<ProjectReference>` 가능 | MSBuild 프로젝트 참조 **불가** |

Engine / Application은 MSVC용 `.vcxproj` Static Library로, ARM용 Clang으로 컴파일해야 하는 Android에서는 MSVC `.lib`를 그대로 링크할 수 없음. VS가 이를 인식하여 경고를 표시하는 것.

### 해결 방향

**단기 (현재 — CMake 미도입):**
ExecAndroid의 참조 설정은 건너뜀. Windows 파이프라인(`ExecWindows → Application → Engine`) 먼저 완성.

**장기 (CMake 도입 후):**
```
Engine/CMakeLists.txt
Application/CMakeLists.txt
ExecAndroid/app/src/main/CMakeLists.txt  ← add_subdirectory로 포함
```
`app/build.gradle`에 아래 설정 추가 시, Gradle이 NDK Clang으로 Engine + Application을 함께 빌드:
```groovy
android {
    defaultConfig {
        externalNativeBuild {
            cmake { path "src/main/CMakeLists.txt" }
        }
    }
}
```

### 요약
- ExecAndroid의 Engine / Application 참조 경고는 정상이며 제거 불가
- Android 네이티브 연동은 CMake 전환 시점에 함께 진행
- 현재는 ExecWindows 참조 설정에 집중

## Q2: IExecute.h에 전역 함수 선언 혼용 시 문제점

**질문:**
`CreateApplication()` 선언을 `IExecute.h` 하단에 넣을 경우 발생하는 문제점.

**답변:**

### 문제 1 — 의존성 방향 오염
`CreateApplication()`은 Application이 정의하는 함수. 선언이 Engine의 `IExecute.h`에 있으면 Engine이 Application 쪽 팩토리 함수의 존재를 암묵적으로 가정하게 됨. **"Engine은 Application을 모른다"** 원칙에 어긋남.

### 문제 2 — Python 자동 생성 스크립트 오작동
`ApplicationFactory.h`가 `Engine/Public/` 하위 디렉토리에 있으면, 스크립트가 `Engine/Private/`에 stub cpp를 자동 생성. 실제 구현은 `Application/`에 있으므로 **링커 중복 정의 오류** 발생.

스크립트는 Engine 프로젝트 범위만 관리하므로 Application 프로젝트의 cpp 존재 여부를 알 수 없음.

### 결론
인터페이스 파일에 전역 함수 선언을 혼용하지 않는다. `CreateApplication()` 구조 자체를 재검토 필요.

## Q3: IExecute 배치 및 플랫폼별 구현 구조 확정

**질문:**
`IExecute`의 메서드들이 어디서 호출되는지, 어디에 배치되어야 하는지.

**답변:**

### 호출 구조
`OnInitialize()`, `OnUpdate()`, `OnRender()` 등은 플랫폼 Loop에서 호출됨. Engine이 호출하는 구조가 아님.

```
ExecWindows (WinMain 메시지 루프)
  → app->OnInitialize()
  → app->OnUpdate()
  → app->OnRender()
  → app->OnPostRender()
  → app->OnDestroy()
```

### 확정 구조

```
Application/
└── IExecute.h        ← 순수 인터페이스 (Engine이 아닌 Application 소유)

ExecWindows/
└── WindowsApplication : public IExecute
      OnInitialize()      ← HWND 등 Windows 리소스 직접 사용
      OnUpdate() / OnRender() / ...

ExecAndroid/
└── AndroidApplication : public IExecute
      OnInitialize()      ← ANativeWindow 등 Android 리소스 직접 사용
      OnUpdate() / OnRender() / ...
```

### 핵심 근거

| 항목 | 이유 |
|---|---|
| `IExecute` → Application 소유 | Loop 호출 주체가 Platform이므로 Engine이 알 필요 없음 |
| 구현체 → 각 Exec 프로젝트 | HWND / ANativeWindow가 이미 해당 프로젝트에 존재, 외부에서 전달 불필요 |
| Engine 무관 | Engine은 렌더링 인프라(IRenderDevice 등)만 제공 |

### 주의
파일 이동은 반드시 VS / Rider IDE에서 수행. 솔루션 파일 경로 참조가 자동 갱신되지 않으면 빌드 오류 발생.
