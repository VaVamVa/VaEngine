#### 3. 구현부 자동 생성기 도입 (이전 프로젝트 이식)
이전 프로젝트(`:\Project\DX12\VaVamVa`)에서 사용하던 Python 기반 빌드 자동화 스크립트를 VaEngine에 이식.

**생성된 파일:**

| 파일 | 역할 |
|---|---|
| `ThirdParty/Main.py` | 진입점. Engine 프로젝트 디렉토리를 찾아 두 함수 호출 |
| `ThirdParty/GeneratePublicHeader.py` | 핵심 로직 2개 포함 |
| `Engine/Public/InnerDependencies.h` | 자동 관리되는 Public 헤더 집약 파일 (stub) |

**GeneratePublicHeader.py 핵심 기능:**

- `update_inner_dependencies()` : `Engine/Public/` 하위 디렉토리 스캔 → `InnerDependencies.h`에 섹션별 `#include` 자동 갱신. 변경 없으면 파일 미갱신 (증분 빌드 보호)
- `create_missing_cpp_files()` : `Public/{subdir}/*.h` 기준으로 `Private/{subdir}/*.cpp` 없으면 자동 생성. **`I~~` 파일(순수 가상 인터페이스) 제외** (이전 프로젝트 대비 추가된 수정)
- `add_generated_cpp_to_vcxproj()` : 생성된 .cpp를 `Engine.vcxproj`의 `<ItemGroup Label="Definitions">`에 자동 등록

**이전 프로젝트 대비 변경점:**

| 항목 | 이전 프로젝트 | VaEngine |
|---|---|---|
| PCH 파일 | `framework.h` | `pch.h` |
| `I~~` 파일 처리 | 없음 (cpp 생성 시도) | 건너뜀 (인터페이스 파일 보호) |

**Engine.vcxproj 변경:**
- 모든 구성에 `AdditionalIncludeDirectories`에 `$(ProjectDir)Public` 추가
- `framework.h`에 `#include "InnerDependencies.h"` 추가 (PCH 체인으로 전파)
- MSBuild Target: `BeforeTargets="ClCompile"`, `Public/*.h` 변경 시에만 실행

**동작 흐름:**
```
빌드 시작
  → ThirdParty/Main.py 실행
      → Public/ 하위 디렉토리 스캔
      → InnerDependencies.h 갱신
      → 누락된 Private/*.cpp 생성 + vcxproj 등록
  → ClCompile (C++ 빌드)
```

**적용 범위:** Engine 프로젝트 전용. Application/ExecWindows는 별도 판단 필요.

#### 4. Engine/Public/Interfaces/IExecute.h 작성 완료

### 요약 내용

#### 5. IExecute 아키텍처 재설계 및 확정

**변경 전 (초기 계획):**
- `IExecute.h` → `Engine/Public/Interfaces/`
- `CreateApplication()` → Engine에 선언, Application에 정의

**변경 후 (확정):**
- `IExecute.h` → `Application/` 프로젝트로 이동
- 플랫폼별 구현체 → 각 Exec 프로젝트에 위치

**변경 이유:**
1. Loop 호출 주체가 Platform이므로 Engine이 IExecute을 알 필요 없음
2. 구현체가 각 Exec 프로젝트에 있으면 HWND / ANativeWindow를 외부에서 전달받을 필요 없이 직접 사용 가능
3. `CreateApplication()` 선언/정의 분리 시 Python 스크립트 중복 stub 생성 위험 제거

**확정 구조:**
```
Application/
└── IExecute.h             ← 순수 인터페이스

ExecWindows/
└── WindowsApplication : public IExecute

ExecAndroid/
└── AndroidApplication : public IExecute
```

    - `IExecute` 구현체
    - WinMain에서 인스턴스 생성 → Loop 호출

    - `WindowsApplication` 생성 → `OnInitialize()` → 메시지 루프 → `OnDestroy()`

#### 6. CMake 파일 수집 방식 확정

- **방식 A (GLOB_RECURSE + CONFIGURE_DEPENDS)** 채택
- 파일 추가 시 CMakeLists.txt 수동 수정 불필요. CMake가 빌드 시 파일 목록 변경 자동 감지
- 이전 Python 스크립트의 `add_generated_cpp_to_vcxproj()` 역할을 CMake가 대체

#### 7. RHI 백엔드 아키텍처 확정

**확정 구조:**
- Engine이 DX12 / Vulkan 소스를 모두 소유
- 동적 라이브러리(DLL) 분리 없음
- `CMakePresets.json` + `option(USE_DX12)` / `option(USE_VULKAN)` 으로 패키징 시 백엔드 선택

**플랫폼별 패키징:**

| 대상 | USE_DX12 | USE_VULKAN |
|---|---|---|
| Windows | ON | OFF |
| Android | OFF | ON |
| Linux | OFF | ON |

**근거:**
- 데스크탑에서 코드 작성, 실제 패키징 단계에서 API 분리로 충분
- 런타임 백엔드 교체는 후순위
- DLL ABI 경계 관리 복잡성 회피

#### 8. FNativeWindowInfo 구조체 설계 확정

- `Engine/Public/` 에 위치. 플랫폼 헤더(`windows.h` 등) 없이 `void*` + `EPlatform` enum만 사용
- `#if` 는 `Engine/Private/RHI/DX12/`, `Engine/Private/RHI/Vulkan/` 구현부에만 존재
- Vulkan: `EPlatform`으로 OS별 Surface 생성 분기 (`VkWin32SurfaceCreateInfoKHR` / `VkAndroidSurfaceCreateInfoKHR` 등)
- DX12: Windows 전용, `HWND` → `IDXGISwapChain` 직접 생성. 플랫폼 분기 없음

### 요약 내용

#### 17. CMakeLists.txt GLOB 패턴 수정

`GLOB_RECURSE`에서 `**` 사용 시 직속 파일을 매칭하지 못하는 문제 확인. `GLOB_RECURSE`가 재귀를 자동 처리하므로 `**` 불필요. 전체 CMakeLists.txt에서 `**` 제거:

| 파일 | 변경 전 | 변경 후 |
|---|---|---|
| Engine | `Private/Core/**/*.cpp` 등 | `Private/Core/*.cpp` 등 |
| Application | `Private/**/*.cpp` | `Private/*.cpp` |
| ExecAndroid | `Source/**/*.cpp` | `Source/*.cpp` |

ExecAndroid에 소스 없을 때 `if(NOT EXECANDROID_SOURCES) return() endif()` 가드 추가.

#### 18. CMakePresets.json android 프리셋 툴체인 방식 변경

`CMAKE_SYSTEM_NAME=Android` (방법 3) 사용 시 sysroot 경로 미설정으로 `-lvulkan` 링크 실패. NDK 자체 툴체인 파일로 전환:

- **변경 전**: `CMAKE_SYSTEM_NAME`, `CMAKE_ANDROID_NDK` cacheVariables
- **변경 후**: `toolchainFile` → `${sourceDir}/Platforms/ExecAndroid/ndk/build/cmake/android.toolchain.cmake`

NDK 툴체인 파일이 sysroot 및 라이브러리 경로를 올바르게 설정함.
