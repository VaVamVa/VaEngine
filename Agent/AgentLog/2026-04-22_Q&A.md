## Q1: Public/Private 헤더 분리의 성능 이점

**질문:**
파일 구조를 잡는데 있어서, Public과 Private으로 나누는게 성능상 이점이 있을지. 있다면 어떻게 나누어야 할 지.

**답변:**

### 결론
컴파일 성능 향상이 있음. 런타임 성능과는 무관하고, **빌드 시간**과 **의존성 전파** 문제.

### 핵심 원리
C++에서 `#include`는 텍스트 복사이므로, 헤더가 노출되는 범위가 넓을수록 불필요한 재컴파일 발생.

```
Engine.lib의 내부 구현 헤더가 Game.lib에 노출되면
  → Game의 .cpp 파일들이 Engine 내부 헤더를 전부 파싱
  → Engine 내부 헤더 하나 수정 → Game 전체 재컴파일
```

### 권장 디렉토리 구조

```
Engine/
├── Public/                  ← 외부(Game, Windows)에 노출되는 인터페이스
│   ├── IApplication.h
│   ├── IRenderDevice.h
│   ├── Math/Vector.h
│   └── Core/Handle.h
│
└── Private/                 ← Engine 내부에서만 사용
    ├── RHI/
    │   ├── DX12/DX12Device.h
    │   └── Vulkan/VulkanDevice.h
    ├── Core/MemoryPool.h
    └── Platform/Windows/WindowsWindow.h
```

### CMake 설정

```cmake
target_include_directories(Engine
    PUBLIC  ${CMAKE_CURRENT_SOURCE_DIR}/Public   # Game, Windows도 볼 수 있음
    PRIVATE ${CMAKE_CURRENT_SOURCE_DIR}/Private  # Engine 내부만 볼 수 있음
)
```

### 분류 기준

| Public | Private |
|---|---|
| 인터페이스 (`I~~`) | 구현체 (`DX12Device`, `VulkanDevice`) |
| 외부가 사용하는 타입 (`Handle`, `Vector`) | 내부 헬퍼, 유틸리티 |
| 등록 패턴 함수 선언 | 플랫폼 API 직접 호출부 (`d3d12.h` 포함 파일) |

판단 기준: **"Game.lib 코드가 이 헤더를 `#include` 할 필요가 있는가?"**
- Yes → Public / No → Private

### 실질적 이점

| 이점 | 설명 |
|---|---|
| 빌드 시간 단축 | `DX12Device.h` 수정 시 Game 재컴파일 없음 |
| 의존성 캡슐화 | `d3d12.h`, `vulkan.h`가 Game에 노출되지 않음 |
| API 경계 명확화 | 공개 인터페이스를 디렉토리 구조로 명시 |
| Pimpl 없이도 유사 효과 | Private 구현 헤더를 숨김으로써 ABI 노출 최소화 |

## Q2: RHI 하위 DX12/Vulkan 디렉토리 구조

**질문:**
Vulkan과 DX의 공간을 나누는게 좋을까 아니면 RHI 폴더 하위에 같이 두는게 좋을까?

**답변:**

### 결론
`RHI/` 하위에 함께 두는 것을 권장.

### 권장 구조

```
Private/
└── RHI/
    ├── DX12/
    │   ├── DX12Device.h
    │   ├── DX12CommandQueue.h
    │   └── DX12SwapChain.h
    └── Vulkan/
        ├── VulkanDevice.h
        ├── VulkanCommandQueue.h
        └── VulkanSwapChain.h
```

RHI는 "렌더링 백엔드 추상화 레이어"라는 의미이므로, DX12와 Vulkan은 둘 다 이 레이어의 구현체 → 같은 폴더 아래 묶는 것이 의미상 자연스러움.

### 분리 시 문제

```
Private/
├── DX12/    ← RHI 구현인지, 플랫폼 코드인지 불명확
├── Vulkan/
└── RHI/     ← 그럼 여기엔 뭐가 들어가야 하는가?
```

디렉토리만 봤을 때 역할이 모호해짐.

### RHI vs Platform 구분 기준

| 디렉토리 | 담당 |
|---|---|
| `Private/RHI/DX12/` | GPU 렌더링 API (파이프라인, 버퍼, 셰이더) — `d3d12.h` |
| `Windows.exe 프로젝트 내부` | OS 레벨 (창 생성, 입력, Win32 메시지 루프) — `windows.h` |

## Q3: Platform 폴더가 Engine 내부에 있어야 하는가

**질문:**
Platform이 왜 Engine 프로젝트 내부에 존재해? Executable 프로젝트로 빼기로 한거 아니야?

**답변:**

### 결론
Engine 내부에 `Platform/` 폴더를 두는 것은 확정 아키텍처와 모순. Agent의 초기 제안 오류.

### 이유

Q10 확정 아키텍처의 핵심 원칙은 **Engine.lib = 플랫폼 무관**.

```
Windows.exe  ← Win32 창 생성, 메시지 루프 (여기에 platform 코드)
Android      ← ANativeActivity, 안드로이드 생명주기 (여기에 platform 코드)
                    ↓ 둘 다 링크
              Engine.lib  ← 플랫폼 코드 없음
```

Engine 안에 `Platform/Windows/`가 있으면 `windows.h`, Win32 API가 Engine에 침투 → "Engine은 플랫폼 무관"이라는 원칙을 스스로 깨는 구조.

### 올바른 Engine 내부 구조

```
Engine/
├── Public/
│   ├── IApplication.h
│   ├── IRenderDevice.h
│   └── Math/
└── Private/
    ├── RHI/
    │   ├── DX12/
    │   └── Vulkan/
    └── Core/     ← 플랫폼 무관한 내부 구현 (메모리 등)
```

`Platform/` 폴더는 Engine이 아닌 각 Executable 프로젝트(`Windows.exe`, `Android`) 안에 위치해야 함.

## Q4: 플랫폼별 프로젝트 실행 라이프사이클

**질문:**
위의 구조의 프로젝트를 실행한다 하였을 때, 예상되는 라이프사이클을 알려줄 수 있어?

**답변:**

### Windows.exe

```
WinMain()
├─ Engine 초기화 → DX12RenderDevice 생성
├─ CreateApplication() → MyGame 생성
├─ OnInit()
└─ 메시지 루프
    ├─ [매 프레임] OnUpdate() → OnRender()
    └─ WM_DESTROY → OnDestroy() → 종료
```

### Android (NativeActivity)

Android는 Surface 생명주기가 별도로 존재하여 Windows보다 복잡.

```
ANativeActivity_onCreate()
├─ Engine 초기화 → VulkanRenderDevice 생성
├─ CreateApplication() → MyGame 생성
├─ onStart() → onResume()
├─ onNativeWindowCreated()   ← Surface 준비, 여기서부터 렌더링 가능
│   └─ OnInit()
├─ 메인 루프 (ALooper_pollAll로 이벤트 처리)
│   ├─ [매 프레임] OnUpdate() → OnRender()
│   ├─ 백그라운드 전환: onPause() → onNativeWindowDestroyed() (렌더링 중단)
│   └─ 복귀: onNativeWindowCreated() → onResume()
└─ onStop() → onDestroy() → OnDestroy() → 종료
```

### 핵심 차이점

| | Windows | Android |
|---|---|---|
| 진입점 | `WinMain` | `ANativeActivity_onCreate` |
| 렌더링 시작 조건 | 창 생성 즉시 | `onNativeWindowCreated` 이후 |
| 백그라운드 전환 | `WM_ACTIVATE` | Surface 소멸 → 렌더러 일시 해제 필요 |
| 이벤트 처리 | `GetMessage` / `PeekMessage` | `ALooper_pollAll` |

### 설계 시 주의

`IApplication` 인터페이스에 `OnSuspend()` / `OnResume()` 훅이 필요할 수 있음.
Android의 Surface 소멸/재생성은 Windows에 없는 고유 상태이므로, 지금 인터페이스 설계 시 반영 여부를 결정해야 함.

## Q5: 최종 빌드 및 실행 워크플로우

**질문:**
Engine 프로그래머는 Engine 프로젝트를, 클라이언트 프로그래머는 Game 프로젝트를 수정할 것인데, 최종적으로 앱을 확인하는 과정은 Platform 프로젝트를 수동으로 빌드하는 것인가?

**답변:**

### 결론
수동 빌드 불필요. VS의 **프로젝트 참조(Project Reference)** 설정으로 F5 하나로 자동 처리됨.

### 동작 방식

`Windows.exe`를 시작 프로젝트(Startup Project)로 설정 시, F5로 의존성 순서대로 자동 빌드 후 실행.

```
F5 (Windows.exe가 시작 프로젝트)
├─ Engine.lib 변경 있음? → 재빌드
├─ Game.lib 변경 있음?   → 재빌드 (Engine.lib 먼저)
├─ Windows.exe 링크
└─ 실행
```

변경되지 않은 프로젝트는 증분 빌드로 스킵.

| 수정한 사람 | 재빌드 대상 |
|---|---|
| Engine 프로그래머 | Engine.lib → Game.lib → Windows.exe |
| 클라이언트 프로그래머 | Game.lib → Windows.exe |

### 설정 방법

VS에서 `Windows.exe` 프로젝트 우클릭 → **참조 추가(Add Reference)** → `Engine`, `Game` 선택.

### Android

동일한 원리. Gradle이 빌드 → APK 패키징 → 디바이스/에뮬레이터 배포까지 자동 수행.
Android 프로젝트를 시작 프로젝트로 설정하면 F5로 동일하게 동작.

## Q6: 모든 프로젝트를 하나의 솔루션으로 관리해야 하는가

**질문:**
이 모든 프로젝트는 하나의 솔루션에서 관리해야 하겠네?

**답변:**

### 결론
맞음. 하나의 솔루션으로 관리해야 프로젝트 참조와 F5 자동 빌드가 작동함.

```
VaEngine.sln
├── Engine    (Static Library)
├── Game      (Static Library)
├── Windows   (Executable)   ← Windows 개발 시 시작 프로젝트
└── Android   (기본 어플리케이션, Gradle)  ← Android 개발 시 시작 프로젝트
```

솔루션이 분리되면 프로젝트 참조가 끊겨 빌드 순서와 링크를 수동으로 관리해야 함.

### 주의점

Android(Gradle) 프로젝트는 빌드 시스템이 달라 Windows 빌드와 동시에 빌드되지 않음.
같은 솔루션 안에 있되, **플랫폼별로 시작 프로젝트를 전환**하며 사용하는 방식.
