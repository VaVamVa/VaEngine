## Q1. Forward Renderer와 Deferred Renderer는 완전히 분리된 독립 경로여야 하지 않는가

**Q**: Forward Renderer는 성급한 렌더링, Deferred Renderer는 지연 렌더링으로 서로 다른 패스로 진입하는 것이 아닌지? 서로 연관성이 없어야 하는 것이 아닌지?

**A**:

### 핵심 오해

Forward와 Deferred는 **경쟁 관계가 아니라 보완 관계**다.  
"Deferred Renderer를 쓴다"는 말은 "Forward를 완전히 제거한다"는 뜻이 아니다.

---

### 각 방식이 담당하는 것

| 구분 | Forward Renderer | Deferred Renderer |
|---|---|---|
| **원리** | 각 물체를 그리면서 즉시 조명까지 계산 | Geometry Pass → G-Buffer → Lighting Pass (분리) |
| **장점** | 단순, 투명 처리 자연스러움 | 조명 수가 많아도 비용 선형 증가 없음 |
| **단점** | 조명 × 물체 수 만큼 연산 폭발 | 투명 처리 어려움, G-Buffer 메모리 필요 |

Deferred가 해결하는 문제는 **다수의 조명 × 다수의 불투명 물체** 조합의 비용이다.  
그 외 요소는 여전히 Forward로 처리하는 것이 자연스럽다.

---

### 왜 Deferred 파이프라인에서도 Forward 패스가 필요한가

**투명 물체 (Transparent)**

G-Buffer는 픽셀당 하나의 표면만 기록한다. 투명 물체는 여러 레이어를 블렌딩해야 하므로 G-Buffer 방식으로는 표현 불가능하다. 반드시 별도 Forward 패스로 처리해야 한다.

**스카이박스 (Sky)**

스카이는 깊이값 1.0 (무한히 멀리)으로 처리되어 G-Buffer Geometry Pass에 기록할 geometry가 없다. Deferred Lighting 셰이더에서 depth ≥ 1.0인 픽셀은 스킵(sky 영역이므로)하도록 구현했고, 스카이 자체는 Forward 방식으로 hdrOut에 미리 그려둔다.

**Debug 드로우 (DebugLine, DebugText)**

조명 계산이 필요 없는 단순 오버레이. Deferred를 거칠 이유가 없다.

**스키닝 메시 (Animation)**

현재는 Forward로 처리 중. 추후 G-Buffer에 통합 가능하지만 스키닝 셰이더의 복잡도를 G-Buffer PSO에 반영해야 한다.

---

### 업계 표준 — Hybrid Deferred + Forward

현대 엔진은 대부분 이 방식을 사용한다.

```
[Deferred Path]
  Geometry Pass   → G-Buffer (불투명 물체)
  Lighting Pass   → HDR 출력 (Compute)
  Sky Pass        → HDR 출력 (Forward)

[Blit]
  HDR → Backbuffer

[Forward Path — 후처리]
  Transparent Pass → Backbuffer
  Debug Pass       → Backbuffer
```

Unreal Engine의 렌더링 파이프라인도 동일한 구조다. BasePass(Deferred)로 불투명 처리 후 Translucency Pass(Forward)를 별도로 실행한다.

---

### 현재 VaEngine 구조에서의 역할 분리

| 클래스 | 역할 |
|---|---|
| `GBufferRenderer` | Deferred 불투명 Geometry Pass |
| `DeferredLightingRenderer` | Deferred Lighting Compute Pass |
| `ForwardRenderer` | Sky + Transparent + Debug (Forward 전담) |
| `AnimationRenderer` | 스키닝 Geometry (현재 Forward, 추후 GBuffer 통합 가능) |
| `SceneRenderer` | 위 4개를 올바른 순서로 조합하는 오케스트레이터 |

`ForwardRenderer`라는 이름이 혼란스러울 수 있지만, 실제 역할은 **"Deferred가 처리할 수 없는 요소들의 Forward 패스 담당자"** 다.

---

### 완전히 분리된 구조가 필요한 경우

만약 `USE_FORWARD` / `USE_DEFERRED` 플래그로 전체 파이프라인을 교체하는 설계라면 분리가 맞다.  
그러나 그 경우에도 Deferred 경로 내에서 투명 처리를 위한 Forward 패스는 반드시 존재한다.  
즉, **"Deferred 렌더러라도 내부에 Forward 패스를 포함한다"** 는 구조적 사실은 변하지 않는다.
