## 리팩터링 대상 분석

### 현재 구조 (원본 DX 프로젝트 기준 + Vulkan 추가 후)

```
Engine          ← RHI 인터페이스 (IRenderDevice 등) + DX12/Vulkan 구현
                   IExecute.h 도 여기에 위치 (문제)
Application     ← 사실상 빈 스텁 (Application.cpp 더미만 존재)
ExecWindows     ← WinMain (ExecWindows.cpp)
                   + Execute.h/.cpp (IExecute 구현체 — 여기 있으면 안 됨)
                      - MatUtil 행렬 유틸리티 (Engine 소속이어야 함)
                      - CUBE_VERTICES / CUBE_INDICES (Engine 소속이어야 함)
                      - 회전·색상 등 씬 로직 (Application 소속이어야 함)
                      - HWND / HINSTANCE 멤버 직접 보유 (플랫폼 타입 노출)
```

### 목표 구조

```
Engine          ← RHI 구현 + 수학(Math) + 기하(Geometry) + 물리(Physics)
Application     ← IExecute 인터페이스 정의 + Execute 구현체 (씬·오브젝트 조작)
ExecWindows     ← WinMain + 플랫폼 루프만 (Execute 생성·호출)
ExecAndroid     ← NativeActivity + 플랫폼 루프만
```

## 아키텍처 결정 사항

### IExecute 위치 변경 이유
- Engine은 렌더링 인프라(RHI)만 알아야 함
- 앱 라이프사이클(OnInitialize/OnLoop 등)은 Application 계층의 계약
- Application이 Engine에 link(PUBLIC)하므로 ExecWindows는 Application 경유로 Engine 타입도 접근 가능

### Execute의 HWND/HINSTANCE 제거 이유
- Execute가 Application에 위치하면 플랫폼 헤더(windows.h) 없이 컴파일되어야 함
- `NativeDisplayInfo`는 `void*`로 추상화되어 있으므로 여기서 꺼낸 값을 로컬 변수로만 사용하고 멤버로는 저장하지 않음
- 또는 멤버로 저장 시 `void* nativeHandle`, `void* nativeDisplay` 형태로 보관
