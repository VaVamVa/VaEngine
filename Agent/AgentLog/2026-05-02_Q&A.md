## Q8: DX12 개요 — DX11 대비 핵심 변화 및 초기화 흐름

**질문:**
DX11 경험은 있으나 DX12가 생소함. RHI 구현 전 DX12 전체 개요 가이드.

**답변:**

### DX11 → DX12 핵심 패러다임 변화

DX12는 DX11이 자동으로 해주던 것들을 **개발자가 직접 관리**하는 방향으로 바뀌었습니다.

| 개념 | DX11 | DX12 |
|---|---|---|
| 커맨드 제출 | `DeviceContext`에 즉시 실행 | `CommandList`에 기록 → `CommandQueue`로 제출 |
| GPU 동기화 | `Present()`가 자동 처리 | `Fence`로 개발자가 직접 관리 |
| 리소스 상태 | 드라이버가 자동 추적 | `ResourceBarrier`로 명시적 전환 |
| 파이프라인 상태 | 각 상태 개별 바인딩 | `PSO` 하나로 묶어서 설정 |
| 리소스 뷰 | `CreateXXXView()` 직접 생성 | `DescriptorHeap`에서 핸들로 관리 |
| 셰이더 바인딩 | `VSSetConstantBuffers` 등 | `RootSignature`로 레이아웃 선언 후 바인딩 |

---

### DX12 초기화 전체 흐름

```
① DXGI Factory 생성
      ↓
② Adapter(GPU) 열거
      ↓
③ D3D12Device 생성
      ↓
④ CommandQueue 생성   ← DX11의 DeviceContext가 여기로
      ↓
⑤ SwapChain 생성      (DXGI — DX11과 구조 유사)
      ↓
⑥ Descriptor Heap 생성   ← DX11에 없던 개념
      ↓
⑦ RTV / DSV 생성 (Descriptor Heap 안에)
      ↓
⑧ CommandAllocator 생성   ← 프레임 수만큼
      ↓
⑨ CommandList 생성
      ↓
⑩ Fence + 이벤트 생성   ← CPU/GPU 동기화
```

---

### 핵심 개념별 설명

#### CommandList / CommandQueue

DX11의 `DeviceContext`가 역할별로 분리됨:

```
CommandList   → "명령을 기록하는 노트"  (Record)
CommandQueue  → "노트를 GPU에 제출하는 창구"  (Execute)
```

CommandList는 `Close()` 후 수정 불가. 재사용하려면 `Reset()` 필요.

| 타입 | 용도 |
|---|---|
| `D3D12_COMMAND_LIST_TYPE_DIRECT` | 렌더링 (DX11 DeviceContext와 동일 역할) |
| `D3D12_COMMAND_LIST_TYPE_COMPUTE` | 컴퓨트 셰이더 전용 |
| `D3D12_COMMAND_LIST_TYPE_COPY` | 리소스 복사 전용 |

#### Fence — 가장 중요한 변화

DX11 `Present()`는 암묵적으로 GPU가 끝날 때까지 기다렸음.
DX12는 **개발자가 직접** CPU-GPU 동기화를 관리:

```
CommandQueue->ExecuteCommandLists()   ← GPU에 작업 제출
CommandQueue->Signal(fence, value)    ← GPU 완료 시 fence 값 증가 예약
fence->GetCompletedValue() < value    ← CPU에서 완료 여부 폴링
WaitForSingleObject(event, INFINITE)  ← 완료 전까지 CPU 대기
```

DoubleBuffering 기준으로 프레임마다 Fence 값을 증가시키며 추적.

#### Descriptor Heap

DX11은 `CreateRenderTargetView()`로 뷰를 직접 생성.
DX12는 **Descriptor Heap이라는 배열**에 뷰를 넣고 인덱스(핸들)로 참조:

```
DescriptorHeap[0] → BackBuffer[0]의 RTV
DescriptorHeap[1] → BackBuffer[1]의 RTV
```

| Heap 타입 | 용도 |
|---|---|
| `RTV` | Render Target View |
| `DSV` | Depth Stencil View |
| `CBV_SRV_UAV` | 상수버퍼, 텍스처, UAV |
| `SAMPLER` | 샘플러 |

#### ResourceBarrier

DX12는 리소스 상태를 드라이버에게 명시적으로 알려줘야 함:

```
프레임 시작: Present    → RenderTarget  (백버퍼를 그리기 모드로)
프레임 끝:   RenderTarget → Present    (백버퍼를 화면 출력 모드로)
```

#### CommandAllocator

CommandList의 실제 메모리를 소유하는 객체.
**GPU가 사용 중인 Allocator는 Reset 불가** → 프레임 수만큼 생성해서 돌아가며 사용:

```
Frame 0: Allocator[0] → CommandList 기록 → Execute → GPU 처리 중
Frame 1: Allocator[1] → CommandList 기록 → Execute → GPU 처리 중
Frame 0: Fence로 완료 확인 후 → Allocator[0].Reset() 가능
```

---

### 우리 프로젝트 인터페이스 → 구현체 대응

| 인터페이스 | 역할 |
|---|---|
| `IRenderDevice` | Device + DXGI + 전체 초기화/해제 |
| `ICommandQueue` | CommandQueue + ExecuteCommandLists |
| `ICommandList` | CommandList + CommandAllocator (쌍으로 관리) |
| `IFence` | Fence + Event + Signal/Wait |
| `ISwapChain` | SwapChain + Descriptor Heap(RTV) + Present |

### 구현 권장 순서

```
1. IRenderDevice  (다른 객체들이 Device를 받아서 생성)
2. IFence         (가장 독립적)
3. ICommandQueue  (Execute + Signal)
4. ISwapChain     (RTV Heap + Present)
5. ICommandList   (렌더링 명령 기록)
```

---

### 프레임 렌더링 루프 전체 구조 (OnLoop 내부)

```
① WaitForPreviousFrame()             ← Fence로 이전 프레임 GPU 완료 확인
② CommandAllocator->Reset()
③ CommandList->Reset(Allocator)
④ Barrier: Present → RenderTarget
⑤ ClearRenderTargetView()
⑥ ClearDepthStencilView()
⑦ ... 렌더링 명령 기록 ...
⑧ Barrier: RenderTarget → Present
⑨ CommandList->Close()
⑩ CommandQueue->ExecuteCommandLists()
⑪ SwapChain->Present()
⑫ CommandQueue->Signal(Fence, ++fenceValue)
```

## Q1: IExecute 구현체 명명 — ExecuteInternal vs WindowsApplication

**질문:**
IExecute 구현체 이름을 `WindowsApplication` → `ExecuteInternal`로 바꾸는 것에 대한 피드백.

**답변:**

### 결론
`WindowsApplication` 유지 권장.

### 근거

| 이름 | 평가 |
|---|---|
| `WindowsApplication` | 플랫폼과 역할이 이름에서 명확히 드러남 |
| `ExecuteInternal` | "내부적으로 실행되는 무언가" — 역할 불명확 |
| `WinApplication` | 짧고 관용적, 대안으로 무난 |
| `WinExecute` | IExecute 계열임을 암시 |

`Internal`은 NVI 훅의 `Impl_` 접두어와 같은 뉘앙스라 혼동 여지가 있음.
`Windows` 접두어를 유지하면 `AndroidApplication` 등 다른 플랫폼 추가 시 일관성도 확보됨.

## Q2: FNativeDisplayInfo 구조체 — IExecute.h에 정의하는 이유

**질문:**
`gExecute = new WindowsApplication(hWnd, hInstance)` 를 채우기 위한 구조체 설계.
생성자는 비우고, 구조체를 `IExecute.h`에 정의하는 방식.

**답변:**

### 구조체 정의 (Engine/Public/Interfaces/IExecute.h)

```cpp
struct FNativeDisplayInfo
{
    enum class EPlatform { Windows, Android, iOS, MacOS };

    EPlatform Platform;

    union
    {
        void*     Handle;   // HWND (Windows) / ANativeWindow* (Android)
        uintptr_t RawData;
    };

    void* Display;          // HINSTANCE (Windows) / nullptr (Android)
};
```

### IExecute.h에 배치한 이유
- `void*` + `EPlatform` enum만 사용 → 플랫폼 헤더(`windows.h` 등) 불필요
- `Engine/Public/` 에서 플랫폼 종속 없이 정의 가능
- `#if` 는 `Engine/Private/RHI/` 구현부에만 존재

### WinMain에서 사용

```cpp
FNativeDisplayInfo displayInfo;
displayInfo.platform = FNativeDisplayInfo::EPlatform::Windows;
displayInfo.Handle   = hWnd;
displayInfo.Display  = hInstance;
execEngine->OnInitialize(displayInfo);
```

## Q3: IExecute* 대신 unique_ptr 사용

**질문:**
`IExecute* gExecute`를 모던 C++ 스마트 포인터로 대체할 수 있는지.

**답변:**

### 결론
`std::unique_ptr<IExecute>` 사용. WinMain이 단독 소유하므로 `unique_ptr`이 적합.

### 변경 사항

```cpp
// 선언
std::unique_ptr<IExecute> gExecute;

// 생성
gExecute = std::make_unique<Execute>(/* args */);

// 해제
gExecute->OnDestroy();
gExecute.reset();   // delete 불필요
```

### 주의
`unique_ptr` 소멸자는 `OnDestroy()`를 자동 호출하지 않음.
`reset()` 전에 반드시 `OnDestroy()` 명시 호출 필요.

WndProc의 `if (gExecute)` 조건은 `unique_ptr`의 `bool` 변환으로 그대로 사용 가능.

## Q4: GetMessage vs PeekMessage — 렌더링 루프 선택 기준

**질문:**
기존 VS 템플릿의 `GetMessage` 루프를 `PeekMessage`로 교체하는 이유.

**답변:**

### 결론
엔진/게임 루프에는 `PeekMessage` 사용.

| 함수 | 동작 | 적합한 용도 |
|---|---|---|
| `GetMessage` | 메시지 없으면 블로킹 (CPU 양보) | 이벤트 기반 일반 앱 |
| `PeekMessage` | 메시지 없으면 즉시 반환 | 렌더링 루프, 게임, 엔진 |

### 구조

```cpp
while (true)
{
    if (PeekMessage(&msg, nullptr, 0, 0, PM_REMOVE))
    {
        if (msg.message == WM_QUIT) break;
        TranslateMessage(&msg);
        DispatchMessage(&msg);
    }
    else
    {
        execEngine->OnLoop();   // 메시지 없을 때 렌더링
    }
}
```

## Q5: 창 제목이 "V"로만 표시되는 원인

**질문:**
`CreateWindowExW`에 `L"VaEngine"`을 전달했는데 창 제목이 "V"로만 표시됨.

**답변:**

### 원인
`UNICODE` / `_UNICODE` 미정의 상태에서 Windows SDK가 Wide 문자열을 ANSI로 해석.

`L"VaEngine"`의 메모리 구조 (UTF-16 LE):
```
'V' = 0x56 0x00
'a' = 0x61 0x00
...
```
ANSI로 읽으면 `0x56`('V') 다음 `0x00`을 null terminator로 인식 → "V"에서 종료.

컴파일 커맨드에 `/D _MBCS`만 있고 `/D UNICODE`가 없는 것이 원인.

### 해결

`framework.h` (`<windows.h>` 앞):
```cpp
#define UNICODE
#define _UNICODE
```

`ExecWindows/CMakeLists.txt`:
```cmake
target_compile_definitions(ExecWindows PRIVATE UNICODE _UNICODE)
```

## Q6: C2850 — PCH 에러 원인 및 해결

**질문:**
`cmake_pch.cxx: Error C2850` — PCH 헤더 파일을 파일 범위에서만 사용할 수 있다는 에러.

**답변:**

### 최종 원인
`Execute.h` 클래스 정의 끝에 `;` 누락.

세미콜론 누락 → 파싱 오류 → PCH 컴파일 단계에서 C2850으로 표출.

### 중간 과정에서 확인한 사항

CMake가 생성하는 `cmake_pch.hxx`:
```cpp
#ifdef __cplusplus
#  include "pch.h"
#endif
```
`#pragma once`가 `#ifdef` 중첩 블록 안에 있으면 C2850이 발생할 수 있음.
그러나 실제 원인은 `;` 누락이었으므로, `#pragma once`는 각 헤더에 그대로 유지.

## Q7: CMake reload — 파일 추가 시마다 필요한지

**질문:**
새 파일을 만들 때마다 CMake reload를 해줘야 하는지.

**답변:**

### 결론
빌드 자체는 자동, IDE 인덱싱은 필요 시에만.

### 근거
프로젝트 CMakeLists가 `CONFIGURE_DEPENDS`를 사용:

```cmake
file(GLOB_RECURSE EXECWINDOWS_SOURCES
    CONFIGURE_DEPENDS
    "*.cpp" "*.rc"
)
```

빌드 시점에 파일 변경을 자동 감지하므로 매번 수동 reload 불필요.

| 상황 | reload 필요 여부 |
|---|---|
| 실제 빌드 | 불필요 (CONFIGURE_DEPENDS가 자동 감지) |
| Rider 자동완성/코드 분석 오류 | 필요 시 1회 |

## Q9: IRenderDevice 설계 — Factory 패턴 및 책임 분리

**질문:**
IRenderDevice가 초기화 진입점이면서 다른 RHI 객체의 Factory 역할을 해야 하는가? Device와 SwapChain의 초기화 책임은 어떻게 분리하는가?

**답변:**

### 결론
IRenderDevice = GPU 초기화 + RHI 객체 Factory. SwapChain은 CreateSwapChain 시점에 윈도우 정보 수신.

### Device vs SwapChain 책임 분리

| 객체 | 초기화 시점 | 필요 정보 |
|---|---|---|
| `IRenderDevice` | `Initialize()` | 없음 (GPU 그 자체) |
| `ISwapChain` | `CreateSwapChain(desc)` | HWND, 해상도, 포맷 등 |

`IRenderDevice::Initialize()`에서 `FNativeDisplayInfo`를 받지 않는 이유:
- Device는 GPU 자체를 제어하는 객체, 특정 윈도우와 무관
- 멀티 윈도우 지원 시 SwapChain을 여러 개 생성 가능
- 헤드리스 렌더링(창 없이 GPU 연산만)도 자연스럽게 지원

### Descriptor 구조체 패턴

```cpp
struct SwapChainDesc {
    uint32_t          width;
    uint32_t          height;
    uint32_t          format;
    uint32_t          bufferCount;
    NativeDisplayInfo displayInfo;  // 윈도우 정보는 여기서
};

virtual std::unique_ptr<ISwapChain> CreateSwapChain(const SwapChainDesc& desc) = 0;
```

인자를 직접 나열하지 않고 구조체로 감싸면 옵션 추가 시 함수 시그니처 변경 불필요.

### Factory 패턴 최종 구조

```cpp
class IRenderDevice {
public:
    virtual void Initialize() = 0;
    virtual void Shutdown()   = 0;

    virtual std::unique_ptr<ISwapChain>    CreateSwapChain(const SwapChainDesc& desc)       = 0;
    virtual std::unique_ptr<ICommandQueue> CreateCommandQueue(const CommandQueueDesc& desc)  = 0;
    virtual std::unique_ptr<IFence>        CreateFence()                                     = 0;
    virtual std::unique_ptr<ICommandList>  CreateCommandList(const CommandListDesc& desc)    = 0;
};
```

DX12 구현체 내부에서 `ID3D12Device*`를 사용해 각 객체를 생성하므로, Public 인터페이스에 API 전용 타입이 노출되지 않음.

### IRenderDevice 자체 생성 — namespace RHI 팩토리

```cpp
// Engine/Public/RHI/RHILoader.h
namespace RHI {
    std::unique_ptr<IRenderDevice> CreateRenderDevice();
}

// Engine/Private/RHI/RHILoader.cpp
std::unique_ptr<IRenderDevice> RHI::CreateRenderDevice() {
#if USE_DX12
    return std::make_unique<DX12RenderDevice>();
#elif USE_VULKAN
    return std::make_unique<VulkanRenderDevice>();
#else
    return nullptr;
#endif
}
```

`class FRHI { static ... }` 대신 `namespace RHI`를 선택한 이유: 정적 멤버만 있는 클래스보다 네임스페이스가 의미적으로 더 자연스러움.

## Q10: DX12 헤더 include 위치 — framework.h vs 각 헤더

**질문:**
`<d3d12.h>`, `<dxgi1_6.h>` 등 DX12 전용 헤더를 `framework.h`에 모아둘지, 각 파일에서 따로 include할지.

**답변:**

### 결론
DX12 전용 헤더는 `Private/RHI/DirectX/` 내부의 공통 헤더(`Common_DirectX.h`)에서만 포함.

### 이유
Engine의 `framework.h`는 PCH를 통해 **모든 Engine 소스에 적용**됨. DX12 헤더를 여기에 넣으면:
- `USE_VULKAN` / Android 빌드에서도 DX12 헤더 포함 → 컴파일 오류
- `Private/RHI/DirectX/`는 `USE_DX12`일 때만 컴파일되므로, 그 안에서만 포함해야 함

### Common_DirectX.h 구조

```cpp
// Engine/Private/RHI/DirectX/Common_DirectX.h
#include <wrl/client.h>    // ComPtr
using Microsoft::WRL::ComPtr;
#include <dxgi1_6.h>       // IDXGIFactory6, IDXGIAdapter4
#include <d3d12.h>         // D3D12 core
#include <d3dx12.h>        // D3DX12 헬퍼 (ThirdParty)
```

DX12 구현 파일들은 이 파일 하나만 include.

## Q11: CMake 헤더 등록 — FILE_SET HEADERS와 IntelliSense

**질문:**
헤더 파일만 존재하는 경우 CMake에 등록되지 않아 IntelliSense가 동작하지 않음. 해결 방법은?

**답변:**

### 원인
Rider IntelliSense는 CMake 프로젝트 모델 기반 동작. `target_sources`에 등록되지 않은 파일은 타겟 컨텍스트(include 경로, defines) 없이 열림 → 심볼 해석 실패.

### 해결 — FILE_SET HEADERS (CMake 3.23+)

```cmake
file(GLOB_RECURSE ENGINE_PUBLIC_HEADERS
    CONFIGURE_DEPENDS
    "Public/*.h"
)

target_sources(Engine PUBLIC
    FILE_SET HEADERS
    BASE_DIRS Public
    FILES ${ENGINE_PUBLIC_HEADERS}
)
```

`GLOB` 패턴을 `FILE_SET FILES`에 직접 쓸 수 없으므로 변수 경유 필수.

### FILE_SET HEADERS 장점

| 항목 | 내용 |
|---|---|
| IDE 가시성 | 프로젝트 트리에 헤더 표시 |
| IntelliSense | 타겟 컨텍스트(include, defines) 적용 |
| install() 연동 | Public 헤더 자동 배포 가능 |
| 의미 명시 | CMake 레벨에서 "Public 헤더"임을 표현 |

헤더를 추가 후 일부러 제거할 이유 없음. 컴파일에 영향 없으므로 영구 유지가 맞음.

## Q12: USE_DX12 — CMake 변수와 C++ 전처리기 매크로 분리

**질문:**
`windows-dx12` 프리셋 선택 중인데 Private/DirectX 헤더가 IntelliSense에 로딩되지 않음.

**답변:**

### 원인
CMake 변수 `USE_DX12`와 C++ 전처리기 `#if USE_DX12`는 **별개의 메커니즘**:

| 메커니즘 | 역할 |
|---|---|
| CMake `if(USE_DX12)` | 어떤 소스 파일을 컴파일에 포함할지 제어 |
| C++ `#if USE_DX12` | 코드 블록 포함 여부 (전처리기) |

CMake 변수가 ON이어도 `target_compile_definitions` 없이는 C++ 코드와 Rider가 `USE_DX12`를 인식하지 못함.

### 해결

```cmake
if(USE_DX12)
    target_compile_definitions(Engine PRIVATE USE_DX12)  # 추가 필요
    ...
endif()

if(USE_VULKAN)
    target_compile_definitions(Engine PRIVATE USE_VULKAN)  # 추가 필요
    ...
endif()
```

추가 후 CMake Reload → Rider가 `#if USE_DX12` 블록을 활성 코드로 인식.

## Q13: DirectX-Headers ThirdParty 통합

**질문:**
로컬에 받아둔 `DirectX-Headers`를 프로젝트에 통합하는 방법.

**답변:**

### 결론
절대 경로 대신 프로젝트 내 `ThirdParty/`로 복사하여 상대 경로로 참조.

### 디렉토리 구조

```
VaEngine/
  ThirdParty/
    DirectX-Headers/
      include/
        directx/
          d3dx12.h
          d3dx12_barriers.h
          d3dx12_core.h
          ... (서브 헤더들)
```

### CMakeLists.txt 적용

```cmake
if(USE_DX12)
    target_include_directories(Engine PRIVATE
        "${CMAKE_SOURCE_DIR}/ThirdParty/DirectX-Headers/include/directx"
    )
endif()
```

`${CMAKE_SOURCE_DIR}` = 루트 `CMakeLists.txt` 위치 → 머신 무관한 상대 경로.

헤더 전용 라이브러리라 용량 작음 (~수백KB). 그대로 커밋 가능.
