## Q1: IExecute 구현체 명명 — ExecuteInternal vs WindowsApplication
**질문:**
IExecute 구현체 이름을 `WindowsApplication` → `ExecuteInternal`로 바꾸는 것에 대한 피드백.

**답변:**
`WindowsApplication` 유지를 권장합니다. 플랫폼과 역할이 이름에서 명확히 드러나며, 추후 `AndroidApplication` 등과의 일관성 확보에도 유리합니다.

## Q2: FNativeDisplayInfo 구조체 — IExecute.h에 정의하는 이유
**질문:**
구조체를 `IExecute.h`에 정의하고 생성자에서 이를 받는 이유.

**답변:**
`void*`와 Enum만 사용하여 플랫폼 헤더 의존성 없이 Engine 레이어에서 정의할 수 있기 때문입니다. 이를 통해 Engine은 플랫폼 무관 원칙을 지키면서도 플랫폼 핸들을 전달받을 수 있습니다.

## Q3: IExecute* 대신 unique_ptr 사용
**질문:**
`IExecute*`를 `unique_ptr`로 대체 가능한가?

**답변:**
네, `std::unique_ptr<IExecute>` 사용을 적극 권장합니다. 다만 `reset()` 전에 `OnDestroy()`를 명시적으로 호출해야 함을 유의해야 합니다.

## Q4: GetMessage vs PeekMessage — 렌더링 루프 선택 기준
**질문:**
왜 렌더링 루프에서는 `PeekMessage`를 사용하는가?

**답변:**
`GetMessage`는 메시지가 없을 때 CPU를 양보하며 대기(Blocking)하지만, `PeekMessage`는 즉시 반환하여 메시지가 없는 유휴 시간에 렌더링 작업을 수행할 수 있기 때문입니다.

## Q5: 창 제목이 "V"로만 표시되는 원인
**질문:**
창 제목이 "V"로만 나오는 현상.

**답변:**
`UNICODE` 매크로 미정의로 인해 Wide 문자열(UTF-16)을 ANSI로 잘못 해석하여 첫 글자 이후의 `0x00`을 문자열 종료로 인식했기 때문입니다. CMake 및 헤더에서 `UNICODE` 정의가 필요합니다.

## Q6: C2850 — PCH 에러 원인 및 해결
**질문:**
C2850 에러 발생 원인.

**답변:**
클래스 정의 끝에 세미콜론(`;`)이 누락되어 PCH 포함 구문이 클래스 내부로 잘못 파싱되었기 때문입니다.

## Q7: CMake reload — 파일 추가 시마다 필요한지
**질문:**
매번 수동으로 리로드해야 하는가?

**답변:**
`CONFIGURE_DEPENDS` 옵션을 사용 중이므로 빌드 시 자동으로 감지됩니다. 다만 IDE 인덱싱 이슈가 있을 때만 수동 리로드를 권장합니다.

## Q8: DX12 개요 — DX11 대비 핵심 변화 및 초기화 흐름
**질문:**
DX12 전체 개요 가이드.

**답변:**
DX12는 리소스 상태 관리, 동기화(Fence), 명령 기록(CommandList) 등을 개발자가 직접 관리해야 합니다. 초기화는 Factory -> Adapter -> Device -> Queue -> SwapChain 순으로 진행됩니다.

---

## Q9: IRenderDevice 설계 — Factory 패턴 및 책임 분리
**질문:**
IRenderDevice의 역할과 SwapChain 초기화 분리.

**답변:**
`IRenderDevice`는 GPU 자체를 관리하며 다른 RHI 객체를 생성하는 Factory 역할을 합니다. 윈도우 종속적인 정보는 `CreateSwapChain` 시점에 `Descriptor`를 통해 전달하여 책임을 분리합니다.

**질문:**

**답변:**
PCH를 오염시키지 않도록 `Private/RHI/DirectX/` 내부의 공통 헤더에서만 포함해야 합니다.

## Q11: CMake 헤더 등록 — FILE_SET HEADERS와 IntelliSense
**질문:**
헤더 파일 인텔리센스 문제 해결.

**답변:**
CMake 3.23+의 `FILE_SET HEADERS`를 사용하여 헤더를 명시적으로 등록하면 IDE 가시성과 인텔리센스 컨텍스트가 확보됩니다.

## Q12: USE_DX12 — CMake 변수와 C++ 전처리기 매크로 분리
**질문:**
CMake 변수를 켰는데 `#if` 블록이 비활성화됨.

**답변:**
CMake 변수와 C++ 매크로는 별개입니다. `target_compile_definitions`를 통해 C++ 전처리기 매크로를 명시적으로 정의해줘야 합니다.

## Q13: DirectX-Headers ThirdParty 통합
**질문:**
외부 SDK 헤더 통합 방법.

**답변:**
절대 경로 대신 `ThirdParty/` 하위에 배치하고 `target_include_directories`를 통해 상대 경로로 참조하여 머신 무관한 빌드 환경을 구축합니다.
