## Q13: std::unique_ptr와 불완전한 형식(Incomplete Type) 오류

**질문:**
`std::unique_ptr<ICommandAlloc>` 멤버를 가진 클래스 빌드 시, `ICommandAlloc`이 불완전한 형식이라며 삭제할 수 없다는 오류(C2027, C2338)가 발생하는 이유는?

**답변:**
`std::unique_ptr`는 객체가 파괴될 때 내부 대상을 `delete` 해야 하므로, 파괴자가 생성되는 시점에 대상 타입의 완전한 정의(크기 및 파괴자 정보)가 필요함. 전방 선언(Forward Declaration)만 되어 있는 상태에서 컴파일러가 파괴자를 생성하려고 하면 이 오류가 발생함.

**해결 방법:**
전방 선언 대신 해당 클래스의 헤더 파일을 직접 `#include` 하여 컴파일러가 전체 정의를 알 수 있게 함으로써 해결함.

## Q15: IResourceView 인터페이스 설계 및 API별 핸들 전달 방식

**질문:**
`IResourceView`를 설계할 때 각 API의 전용 핸들(DX12의 Descriptor, Vulkan의 ImageView)을 어떻게 전달하는 것이 좋은가?

**답변:**
인터페이스는 순수하게 유지하고, 각 API 구현체(`ResourceView_DirectX` 등)가 생성 시점에 필요한 핸들을 직접 받도록 설계함. `void*`를 통해 `Desc` 구조체에 핸들 값을 담아 넘기는 방식이 실무적으로 유연함.

**구조 제안 (sudo-code):**
```cpp
struct ResourceViewDesc {
    EResourceViewType type;
    void* nativeHandle; // DX12의 경우 SIZE_T ptr를 캐스팅하여 보관
};

class ResourceView_DirectX : public IResourceView {
    void Register(Desc, Resource) override {
        this->handle.ptr = reinterpret_cast<SIZE_T>(Desc.nativeHandle);
    }
};
```

## Q16: TRHIResource에서 shared_ptr 사용 시의 메모리 오류와 해결책

**질문:**
`TRHIResource`에서 `std::shared_ptr<ID3D12Resource>`를 사용했을 때 종료 시 `is_block_type_valid` 어설션이 발생하는 이유는?

**답변:**
`ID3D12Resource`는 COM 객체이므로 `delete`가 아닌 `Release()`를 호출해야 함. 일반 `shared_ptr`는 `delete`를 호출하여 힙 손상을 일으킴. 플랫폼 독립성을 유지하며 이를 해결하려면 `shared_ptr` 생성 시 **커스텀 삭제자(Custom Deleter)**를 전달해야 함.

**해결 방식 (sudo-code):**
```cpp
// DX12 구현부에서 생성 시
auto deleter = [](ID3D12Resource* p) { if(p) p->Release(); };
resource = std::shared_ptr<ID3D12Resource>(nativePtr, deleter);
```

## Q17: GPU-CPU 동기화(Fence)의 개념과 효율적인 대기 지점

**질문:**
Fence를 이용한 동기화에서 `Signal`과 `Wait`은 각각 어디서 수행하는 것이 효율적인가?

**답변:**
- **Signal**: 모든 명령 기록과 `Present`가 끝난 직후(`OnLoop` 끝단)에 보내어 "이 프레임 작업 완료"를 알림.
- **Wait**: 다음 프레임의 자원(Allocator 등)을 재사용하기 직전(`OnRender` 시작 지점)에 수행하여, GPU가 일하는 동안 CPU가 다른 작업(Update 등)을 병렬로 처리할 수 있게 함.
- **Timeline Fence**: 64비트 정수 값이 계속 증가하는 방식을 사용하며, 특정 값에 도달했다는 것은 그 이전의 모든 작업도 완료되었음을 의미함.

## Q2: if constexpr의 template Backend는 어디서 정의하는가

**질문:**
`if constexpr` 제안(5.3)에서 `template<ERHIBackend Backend>`를 어디서 정의하는가?

**답변:**
`ERHIBackend` enum과 `kCurrentBackend` 상수를 `Common_RHI.h` 또는 `RHILoader.h`에 정의하고, `kCurrentBackend`의 값은 `#if USE_DX12` 등으로 결정해야 함. 결국 `#if`가 사라지지 않는 구조임.

```cpp
// kCurrentBackend 설정 — 결국 #if 필요
#if USE_DX12
constexpr ERHIBackend kCurrentBackend = ERHIBackend::DirectX12;
#elif USE_VULKAN
constexpr ERHIBackend kCurrentBackend = ERHIBackend::Vulkan;
#endif
```

CMake 플래그(`-DUSE_DX12`)를 소비하는 진입점이 반드시 전처리기여야 하므로 `#if`를 완전히 제거하는 것은 불가능함. 이것이 5.3 제안이 근본적으로 한계를 갖는 이유임.

## Q3: 다중 플랫폼 프로젝트 특성상 template 사용이 불가능한가

**질문:**
다중 플랫폼을 지향하는 프로젝트 특성상, template을 쓰면 플랫폼을 Application 프로젝트에서 지정해야 해서 사용이 불가능한 것인가?

**답변:**
그 추론은 맞지 않음. template에 default 인자를 주면 호출부는 깔끔하게 유지됨.

```cpp
template<ERHIBackend Backend = kCurrentBackend>
std::unique_ptr<IRenderDevice> CreateRenderDevice();

// 호출부에서 플랫폼 명시 없이 사용 가능
renderDevice = RHI::CreateRenderDevice();
```

Application 프로젝트가 플랫폼을 직접 지정하지 않아도 됨. 근본 이유는 다른 곳에 있음 — `kCurrentBackend` 값 자체를 결정할 때 여전히 `#if`가 필요하다는 것.

## Q4: template과 #if 혼용 시 이점이 있는가

**질문:**
`#if`와 `if constexpr` (template 포함)을 혼용해서 쓴다면 이점이 있는가?

**답변:**
실질적인 이점이 하나 있음. **IDE가 비활성 분기를 인식하게 됨.**

현재 `#if USE_DX12`로 가려진 Vulkan 분기는 IDE에서 inactive code로 처리되어 자동완성, 리팩토링, 오류 감지가 동작하지 않음. `if constexpr`로 바꾸면 IDE가 모든 분기를 타입 시스템 안에서 인식함.

단, Vulkan 구현이 없는 상태라면 실익이 없고 코드 복잡도(enum 정의, default template argument, 헤더 이동 또는 명시적 인스턴스화)만 늘어남. **Vulkan 구현이 실제로 존재하고 동시에 유지보수될 때 고려할 선택지.**

## Q5: 엔진 이용자 유지보수 측면에서 어느 방식이 더 좋은가

**질문:**
인터페이스 변경이 있을 때, `#if`로 처리하면 다른 Backend의 이상을 잡아내지 못하고 상수 처리(`if constexpr`)로 하면 잡아낼 수 있다는 것이 맞는가? 완성된 엔진을 이용하는 이용자 유지보수 측면에서 어느 쪽이 더 좋은가?

**답변:**
맞음. 유지보수 이점 자체는 `if constexpr` 혼용 방식에 있음. `#if` 방식은 인터페이스 변경 시 현재 빌드 설정의 백엔드 오류만 표면화됨. `if constexpr` 방식은 비활성 분기도 타입 체크하므로 DX12 빌드 중에도 Vulkan 쪽 컴파일 오류를 즉시 발견할 수 있음.

단, 이 이점은 **엔진을 개발하는 VaVamVa 본인**에게 해당하는 것임. 엔진 사용자는 백엔드 하나를 골라서 쓰므로 다른 백엔드의 오류에 관심이 없음. 두 방식이 서로 다른 대상에게 이점이 있음.

| | 이점 대상 |
|---|---|
| `#if` 단독 | 엔진 **사용자** — SDK 선택적 설치, 빌드 환경 단순 |
| `#if` + `if constexpr` | 엔진 **개발자** — 다중 백엔드 동시 유지보수 |

## Q6: 상수 처리 시 미사용 백엔드 코드도 패키징되는가

**질문:**
`#if`로 처리하면 사용자가 선택한 백엔드만 패키징되는데, 상수 처리(`if constexpr`)를 하면 사용하지 않는 플랫폼의 코드도 같이 패키징되는 것 아닌가?

**답변:**
`if constexpr`의 비활성 분기는 해당 함수 내에서 기계어를 생성하지 않음. `CreateRenderDevice()` 안의 Vulkan 분기는 바이너리에 포함되지 않음.

그러나 **`.cpp` 컴파일 단위가 문제임.** `RenderDevice_Vulkan.cpp` 자체가 CMake 소스 목록에 있으면 `if constexpr`와 무관하게 별도 컴파일 단위로 컴파일되고 링크됨. `if constexpr`는 함수 하나의 코드 생성만 제어할 뿐, 다른 `.cpp` 파일이 링크되는 것을 막지 못함.

현재 `#if` 방식은 CMake에서 소스 파일 자체를 조건부로 포함하기 때문에 이 문제가 없음. `if constexpr`를 쓰더라도 CMake 레벨의 조건부 소스 관리는 그대로 필요함.

## Q7: 완성된 엔진 이용자 관점에서 CMake 소스 관리 + 상수 분기가 더 좋은가

**질문:**
코드 복잡도의 단점만 제외하고 보면, 완성된 엔진을 이용하는 이용자의 유지보수 측면에서는 CMake의 소스 관리와 상수 분기 처리가 더 좋은가?

**답변:**
엔진 **개발자** 관점에서는 맞음. 핵심 이유는 인터페이스 변경 시 모든 백엔드 오류가 단일 빌드에서 동시에 표면화된다는 것.

엔진 **사용자** 관점에서는 `#if` 단독 방식이 더 적합함. `if constexpr` 혼용의 유지보수 이점은 엔진을 개발하는 사람에게 해당하는 이점이며, 사용자 입장에서는 쓰지도 않는 Vulkan SDK를 강제 설치해야 하는 부담만 생김.

## Q8: CMakePreset에 Multi-platform 분기 추가에 대한 피드백

**질문:**
CMakePreset에서 Multi-platform 분기를 하나 더 만드는 것에 대해 어떻게 생각하는가?

**답변:**
좋은 아이디어임. 사용자/개발자 구분 문제를 CMake 프리셋 레벨에서 깔끔하게 분리함.

```
CMakePresets:
├── DX12          → 단일 백엔드 사용자
├── Vulkan        → 단일 백엔드 사용자
└── Multiplatform → 엔진 개발자/CI (모든 백엔드 검증)
```

다만 Multiplatform 빌드에서 `CreateRenderDevice()`가 컴파일 타임에 어떤 백엔드를 선택하는지 설계 결정이 필요함. 현재 구조 확장(컴파일 타임 선택 유지)과 런타임 선택 허용 두 가지 방향이 있으며, VaEngine 규모상 컴파일 타임 선택 유지가 현실적임.

## Q9: 두 USE 매크로를 모두 켜면 if constexpr가 필요 없는가

**질문:**
`USE_DX12`와 `USE_VULKAN`을 모두 ON으로 하면 두 헤더가 모두 포함될 텐데, 이 경우에는 `if constexpr`가 필요 없는 것 아닌가?

**답변:**
절반만 맞음. 두 헤더가 모두 포함되면 두 타입이 **보이기는** 하지만, 컴파일러가 인터페이스 구현 여부를 검증하는 시점은 **인스턴스화할 때**임. `make_unique<RenderDevice_Vulkan>()`을 시도해야 "abstract class라 생성 불가" 오류가 잡힘.

현재 `#elif` 구조에서는 Multiplatform 빌드여도 Vulkan 분기가 코드에 없어서 인스턴스화가 일어나지 않음. `if constexpr`는 non-template 함수에서도 비활성 분기의 semantic analysis를 수행하기 때문에 인스턴스화 없이도 abstract class 오류를 잡아냄.

| | Vulkan 오류 감지 |
|---|---|
| `#elif` + 두 헤더 포함 | X — 인스턴스화 안 됨 |
| 별도 `#if` + 두 헤더 포함 | O — 하지만 unreachable code 발생 |
| `if constexpr` + 두 헤더 포함 | O — 깔끔하게 해결 |

## Q10: Multi-platform 대신 Engine 프리셋으로 변경하는 것에 대한 피드백 및 소스코드 제안

**질문:**
"Multi-platform"이라는 이름이 의미가 너무 애매하므로 차라리 "Engine"이라는 프리셋으로 이름을 바꾸고 두 백엔드를 모두 켜게 하려 한다. 피드백과 소스코드 제안을 해달라.

**답변:**
좋은 결정임. "Multi-platform"은 런타임에 여러 플랫폼을 동시 지원한다는 오해를 줄 수 있음. "Engine"은 엔진 개발자용 빌드라는 의도가 명확하고, 사용자용 프리셋(DX12, Vulkan)과 역할이 분명히 구분됨.

소스코드 제안: `USE_DEBUGGING` 단일 플래그 도입으로 Engine 프리셋을 다른 백엔드 플래그와 독립적으로 관리.

```cpp
// 헤더: 별도 #if로 Engine 프리셋에서 모두 포함
#if USE_DEBUGGING
    #include "DirectX/RenderDevice_DirectX.h"
    #include "Vulkan/RenderDevice_Vulkan.h"
#elif USE_DIRECTX
    #include "DirectX/RenderDevice_DirectX.h"
#elif USE_VULKAN
    #include "Vulkan/RenderDevice_Vulkan.h"
#endif

std::unique_ptr<IRenderDevice> RHI::CreateRenderDevice()
{
#if USE_DEBUGGING
#pragma message("RHI Debugging. Can't Execute Engine")
    { [[maybe_unused]] auto device = std::make_unique<RenderDevice_DirectX>(); }
    { [[maybe_unused]] auto device = std::make_unique<RenderDevice_Vulkan>(); }

#elif USE_DIRECTX
    return std::make_unique<RenderDevice_DirectX>();

#elif USE_VULKAN
    return std::make_unique<RenderDevice_Vulkan>();
#endif

    return nullptr;
}
```

백엔드 추가 시 `USE_DEBUGGING` 블록에 헤더 include와 인스턴스화 블록 하나씩만 추가하면 됨.

## Q11: Engine 프리셋의 generator 및 binaryDir 설정

**질문:**
Engine 프리셋의 `generator`와 `binaryDir`는 어떻게 설정해야 하는가?

**답변:**
`generator`는 다른 프리셋과 동일하게 설정하되, 반복을 피하기 위해 hidden base 프리셋에서 공유하는 것이 좋음. `binaryDir`는 반드시 다른 프리셋과 분리해야 함. CMakeCache가 다른 설정을 가지므로 같은 디렉터리를 공유하면 충돌이 남.

```json
{
  "name": "base",
  "hidden": true,
  "generator": "Visual Studio 17 2022",
  "architecture": { "value": "x64", "strategy": "set" }
},
{
  "name": "engine",
  "displayName": "Engine Debugging",
  "inherits": "base",
  "binaryDir": "${sourceDir}/Build/engine",
  "cacheVariables": { "USE_DEBUGGING": "ON" }
}
```

## Q12: 백엔드가 추가될 때 else 블록에만 추가하면 되는가

**질문:**
사용하는 graphics library가 늘어나면 else 블록에 해당 생성자만 추가하면 되는가?

**답변:**
현재 구현 방식(`USE_DIRECTX && USE_VULKAN` 조합 `#if`)에서는 그렇지 않음. 조합이 폭발적으로 늘어나는 문제가 있음.

```
2개 백엔드: 1가지 조합
3개 백엔드: 4가지 조합 (백엔드 추가마다 2배씩 증가)
```

**확장 가능한 구조**: 조합 대신 각 백엔드를 독립적으로 체크하는 방식을 사용.

```cpp
// USE_DEBUGGING 블록 내부: 각 백엔드 독립 체크
// 백엔드 추가 시 이 블록 하나만 추가
#if USE_DIRECTX
    { [[maybe_unused]] auto tmp = std::make_unique<RenderDevice_DirectX>(); }
#endif
#if USE_VULKAN
    { [[maybe_unused]] auto tmp = std::make_unique<RenderDevice_Vulkan>(); }
#endif
// Metal 추가 시:
// #if USE_METAL
//     { [[maybe_unused]] auto tmp = std::make_unique<RenderDevice_Metal>(); }
// #endif
```

Metal 추가 시 `#if USE_METAL` 블록 하나와 반환 `#elif USE_METAL` 줄 하나만 추가하면 됨. 조합 증가 문제 없음.
