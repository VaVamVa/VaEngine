## Q1: CMake 파일 수집 방식 — 방식 B 수동 추가 여부

**질문:**
방식 B(명시적 target_sources)를 채택할 경우, 파일 추가마다 CMakeLists.txt를 수동 수정해야 하는지.

**답변:**

### 결론
맞음. 방식 B는 파일 추가 시마다 CMakeLists.txt에 직접 항목을 추가해야 함.

### 방식 비교

| | 방식 A (GLOB_RECURSE) | 방식 B (명시) |
|---|---|---|
| 파일 추가 시 | CMake 자동 감지 (`CONFIGURE_DEPENDS`) | CMakeLists.txt 수동 수정 필요 |
| vcxproj 자동 등록 | CMake가 대체 | 직접 관리 |
| 안정성 | 파일 삭제 시 stale 오브젝트 가능성 | 완전한 명시적 제어 |

### 채택
방식 A (`GLOB_RECURSE + CONFIGURE_DEPENDS`) 채택. 이전 Python 스크립트(`add_generated_cpp_to_vcxproj`)가 하던 역할을 CMake가 대체.

## Q2: RHI 백엔드 빌드 분리 — 동적 라이브러리 없이 CMake로 제어 가능한지

**질문:**
동적 라이브러리(DLL) 없이 CMake만으로 플랫폼별 그래픽 API 빌드 분리가 가능한지.

**답변:**

### 결론
가능함. `option()` + `if()` + `CMakePresets.json` 조합으로 빌드 시점에 백엔드를 고정.

### 제어 방식

```
CMakePresets.json
  windows 프리셋 → USE_DX12=ON,  USE_VULKAN=OFF
  android 프리셋 → USE_DX12=OFF, USE_VULKAN=ON
```

Engine이 모든 그래픽 API 소스를 소유하되, 실제 컴파일에 포함되는 소스는 옵션에 따라 결정됨.

### 트레이드오프

| 항목 | 내용 |
|---|---|
| 런타임 백엔드 교체 | 불가 (빌드 시점 고정) |
| DLL ABI 관리 | 불필요 |
| 빌드 복잡도 | 낮음 |

런타임 교체는 후순위이므로 이 방식으로 확정.

## Q3: FNativeWindowInfo — Engine/Public에서 플랫폼 추상화 및 #if 격리

**질문:**
OS에 따른 `#if` 를 Engine 프로젝트에서만 쓸 것 같은데, 아래 구조체로 Engine에 영향을 줄 수 있는지.

```cpp
struct FNativeWindowInfo {
    enum class EPlatform { Windows, Android, iOS, MacOS };
    EPlatform Platform;

    union {
        void* Handle;
        uintptr_t RawData;
    };

    void* Display; // Linux/Android용
};
```

**답변:**

### 결론
가능하고 올바른 방향. `void*` + enum만 사용하므로 플랫폼 헤더 없이 `Engine/Public/` 에 배치 가능.

### #if 분포

| 위치 | #if 존재 여부 |
|---|---|
| `Engine/Public/FNativeWindowInfo.h` | 없음 |
| `Engine/Public/Interfaces/IRenderDevice.h` | 없음 |
| `Engine/Private/RHI/DX12/` | 있음 (`windows.h`, HWND 캐스팅) |
| `Engine/Private/RHI/Vulkan/` | 있음 (각 플랫폼 Surface 헤더) |

CMake 옵션으로 해당 Private 파일 자체가 컴파일에서 제외되므로, Android 빌드에서 `windows.h` 가 아예 들어오지 않는 구조가 됨.

## Q4: Vulkan Surface 생성 방식이 OS별로 다른지

**질문:**
Vulkan의 경우 Windows / Android 외에 Linux도 지원하므로 `EPlatform` 으로 구분이 필요한지.

**답변:**

### 결론
필요함. Vulkan은 OS별로 Surface 생성 구조체가 다름.

| OS | Vulkan Surface 구조체 |
|---|---|
| Windows | `VkWin32SurfaceCreateInfoKHR` (HWND + HINSTANCE) |
| Android | `VkAndroidSurfaceCreateInfoKHR` (ANativeWindow*) |
| Linux (XCB) | `VkXcbSurfaceCreateInfoKHR` |
| Linux (Xlib) | `VkXlibSurfaceCreateInfoKHR` |

같은 Vulkan 구현체 안에서 `EPlatform` 으로 분기하므로 enum이 의미를 가짐.

## Q5: DX12 Surface 생성 방식이 OS별로 다른지

**질문:**
DX12도 Surface 생성 방식이 OS별로 다른지.

**답변:**

### 결론
아님. DX12는 Windows 전용이므로 Surface 개념 자체가 없음.

DX12에서 대응 개념은 `IDXGISwapChain`. 생성에 `HWND` 하나만 필요하며 플랫폼 분기 없음.

## Q6: CMake option 감지 — enum vs #if 매크로

**질문:**
CMake option을 C++에서 감지할 때 enum으로 변화를 줄 수 있는지, 아니면 `#if` 매크로가 더 적합한지.

**답변:**

### 결론
용도가 다름. 백엔드 선택은 `#if` 매크로, OS별 런타임 분기는 `EPlatform` enum.

### CMake option → `#define` → `#if`

`target_compile_definitions`으로 CMake 옵션을 C++ 매크로로 전달:

```cmake
if(USE_DX12)
    target_compile_definitions(Engine PRIVATE USE_DX12)
endif()
if(USE_VULKAN)
    target_compile_definitions(Engine PRIVATE USE_VULKAN)
endif()
```

C++에서 컴파일 시점에 해당 블록 자체가 바이너리에서 제외됨:

```cpp
#if USE_DX12
    // DX12 전용 코드
#endif
```

### EPlatform enum은 별개

Vulkan 구현체 내부에서 런타임에 OS별 Surface 생성 방식을 분기하는 용도. Vulkan 단일 바이너리 안에서 "어떤 OS 핸들인지"를 실행 중에 판단해야 하므로 enum이 맞음.

### 정리

| 용도 | 도구 |
|---|---|
| 어떤 백엔드를 컴파일에 포함할지 | CMake option → `#define` → `#if` |
| Vulkan 내부 OS별 Surface 생성 분기 | `EPlatform` enum (런타임) |

## Q7: Android NDK — Command line tools만으로 설치 가능한지

**질문:**
Android Studio 없이 Command line tools만으로 NDK 설치가 가능한지.

**답변:**

### 결론
가능함. `sdkmanager`로 NDK만 설치하면 됨. Android Studio 불필요.

### 설치 순서

1. Android 개발자 사이트에서 Command line tools (zip) 다운로드
2. 압축 해제 후 `AppData\Local\Android\Sdk\cmdline-tools\latest\` 에 배치
3. sdkmanager로 NDK 설치:
```
sdkmanager "ndk;{버전}"
```
4. 설치 경로: `AppData\Local\Android\Sdk\ndk\{버전}\`
5. `ANDROID_NDK_HOME` 환경변수에 위 경로 등록

### 참고
CMake는 Visual Studio에 번들로 포함되어 있으므로 별도 설치 불필요.

## Q8: CMakePresets android 프리셋 — 로컬 NDK 경로 지정 방법

**질문:**
환경변수 없이 프로젝트 내부 로컬 경로의 NDK를 CMakePresets.json에서 지정하는 방법.

**답변:**

### 결론
`${sourceDir}`를 이용하면 CMakePresets.json 위치 기준 상대 경로 지정 가능.

```json
"CMAKE_ANDROID_NDK": "${sourceDir}/Platforms/ExecAndroid/ndk"
```

`${sourceDir}`는 CMakePresets.json이 위치한 디렉토리(프로젝트 최상위)를 가리킴.

### NDK 버전 폴더 문제
sdkmanager 설치 시 `ndk/{버전}/` 형태로 하위 폴더가 생성됨. CMakePresets.json은 와일드카드 미지원이므로 자동 탐색 불가. 버전 폴더 내용물을 `ndk/` 바로 아래로 이동하여 경로를 고정하는 방식 채택.

### .gitignore 추가
NDK는 용량이 크므로 반드시 ignore:
```
Platforms/ExecAndroid/ndk/
```

## Q9: Android Vulkan — find_package 오류 및 해결

**질문:**
Android 프리셋으로 CMake 구성 시 `Could NOT find Vulkan (missing: Vulkan_LIBRARY)` 오류 발생.

**답변:**

### 원인
`find_package(Vulkan REQUIRED)`는 데스크탑 Vulkan SDK를 탐색하는 명령. Android에서 Vulkan은 OS에 내장된 시스템 라이브러리(`libvulkan.so`)이므로 별도 SDK가 없음.

### 해결
`Engine/CMakeLists.txt`에서 Android / non-Android 분기 처리:

```cmake
if(USE_VULKAN)
    if(ANDROID)
        target_link_libraries(Engine PRIVATE vulkan)
    else()
        find_package(Vulkan REQUIRED)
        target_link_libraries(Engine PRIVATE Vulkan::Vulkan)
    endif()
endif()
```

Android에서는 NDK가 `libvulkan.so` 스텁을 제공하고, 실제 구현은 기기 OS가 담당.

## Q10: 레이어 순서 — Platform → Engine → Application vs Engine ← Application ← Platform

**질문:**
`Platform → Application → Engine` 과 `Platform → Engine → Application` 중 어느 순서가 맞는지.

**답변:**

### 결론
**`Platform → Engine → Application`** 이 맞음.

### 근거

| 항목 | 설명 |
|---|---|
| IExecute 계약 주체 | Engine이 정의하는 생명주기 계약. Application이 소유하면 계약의 근거가 없음 |
| UE 참조 구조 | Engine이 `BeginPlay`, `Tick` 등 생명주기를 정의 → 게임 모듈(Application)이 구현 → Platform이 루프 진입 |
| 의존성 방향 | Platform이 Engine의 IExecute를 호출 → Application은 Engine API를 사용하는 클라이언트 |

### 확정 구조

```
Platform (OS 및 Target Properties 결정)
  → Engine  (Target에 맞는 API 채택, IExecute 정의)
    → Application (Client — 앱 로직, IExecute 구현)
```

### IExecute 위치 변경
- **변경 전**: `Application/Public/Interfaces/IExecute.h`
- **변경 후**: `Engine/Public/Interfaces/IExecute.h`
